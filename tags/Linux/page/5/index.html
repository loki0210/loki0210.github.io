Linux 日志治理：logrotate、journalctl、采集和告警
  • 06/21
  • 15:05

一、日志为什么需要治理

日志是排查线上问题最直接的证据，但不加治理的日志也会带来风险：

  • 文件无限增长，磁盘被写满。
  • 多进程写同一个文件，内容交错。
  • 日志级别混乱，关键错误被淹没。
  • 服务重启后旧日志无法关联。
  • 没有采集和告警，问题只能靠人工发现。

日志治理要解决三个问题：写到哪里、保留多久、怎么发现异常。

  • 06/21

一、为什么要重视内存

Linux 应用运行时间越长，内存问题越容易暴露。短时间测试正常，不代表长期运行稳定。内存泄漏、越界访问、重复释放、use-after-free 都可能导致崩溃、性能下降或系统被 OOM killer 杀死。

常见内存问题：

  • 内存泄漏。
  • 空指针访问。
  • 数组越界。
  • 重复释放。
  • 释放后继续使用。
  • 栈空间不足。
  • 内存碎片。

一、RootFS 是什么

RootFS 是根文件系统，是 Linux 用户态运行的基础。内核启动后，必须挂载 RootFS，然后启动第一个用户态进程。

RootFS 里通常包含：

  • Shell。
  • 基础命令。
  • 动态库。
  • 配置文件。
  • 设备节点。
  • 启动脚本。
  • 应用程序。

一、为什么需要 epoll

阻塞 socket 服务端一次只能等待一个连接或一个读写事件。客户端数量少时问题不大，但连接数变多后，多进程和多线程模型会带来上下文切换、内存占用和调度开销。

Linux 下高并发网络服务常用 epoll。

epoll 适合：

  • 大量连接。
  • 多数连接空闲。
  • 事件驱动服务端。
  • 长连接服务。
