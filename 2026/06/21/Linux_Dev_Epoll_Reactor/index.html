一、为什么需要 epoll

阻塞 socket 服务端一次只能等待一个连接或一个读写事件。客户端数量少时问题不大，但连接数变多后，多进程和多线程模型会带来上下文切换、内存占用和调度开销。

Linux 下高并发网络服务常用 epoll。

epoll 适合：

  • 大量连接。
  • 多数连接空闲。
  • 事件驱动服务端。
  • 长连接服务。

五、水平触发和边缘触发

水平触发 LT：

  • 只要 fd 还有数据可读，epoll_wait 就会继续通知。
  • 编程简单。
  • 入门推荐。

边缘触发 ET：

  • 状态变化时通知一次。
  • 必须配合非阻塞 IO。
  • 必须循环读到 EAGAIN。
  • 性能好，但更容易写错。

九、常见坑

9.1 忘记处理半包和粘包

TCP 是字节流，不保留消息边界。应用层必须设计协议，例如：

  • 固定长度头。
  • 分隔符。
  • length + body。

9.2 阻塞操作放进事件循环

事件循环里不要做耗时操作，例如：

  • 慢磁盘 IO。
  • 复杂计算。
  • 阻塞数据库请求。
  • 长时间 sleep。

否则一个连接会拖慢所有连接。

9.3 没处理对端关闭

read 返回 0 表示对端关闭连接，需要清理 fd。

9.4 边缘触发没有读到 EAGAIN

ET 模式必须循环读，否则可能再也收不到通知。

十、学习路线

建议：

  1. 写阻塞 TCP server。
  2. 写多线程 TCP server。
  3. 写 select server。
  4. 写 epoll LT server。
  5. 改成非阻塞。
  6. 实现 length + body 协议。
  7. 加写缓冲。
  8. 再研究 ET 和线程池。

十一、总结

epoll 不是网络编程的起点，而是进阶工具。先理解 socket、阻塞 IO 和 TCP 字节流，再学习 epoll，会更容易理解它解决了什么问题。

Reactor 的核心是事件循环和状态管理。写对一个稳定的 epoll 服务端，重点不只是 API，而是连接生命周期、缓冲区和错误处理。
