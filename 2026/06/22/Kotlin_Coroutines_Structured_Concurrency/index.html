一、协程解决什么问题

协程不是为了让代码“看起来更高级”，而是为了用同步写法组织异步任务。网络请求、数据库读写、文件 IO、定时任务都可以放进协程里，让代码更容易串联和取消。

传统回调写法容易出现几个问题：

  • 多层回调嵌套。
  • 错误处理分散。
  • 页面销毁后任务还在执行。
  • 多个异步任务之间的生命周期不清楚。

协程的价值是把这些异步任务放到明确的作用域中管理。

二、结构化并发

结构化并发的核心是：协程必须属于某个作用域，父协程结束时，子协程也应该一起结束。

这里 banner 和 articles 是并发执行的，但它们都受 coroutineScope 管理。如果其中一个失败，整个作用域会失败，另一个任务也会被取消。

GlobalScope 的生命周期太长，容易造成页面退出后任务继续执行。

三、常用作用域

Android 中常见作用域：

  • viewModelScope：跟随 ViewModel 生命周期。
  • lifecycleScope：跟随 Activity 或 Fragment 生命周期。
  • rememberCoroutineScope：Compose 中跟随当前组合生命周期。

一般建议：

  • 页面业务请求放在 viewModelScope。
  • 和界面生命周期强相关的动作放在 lifecycleScope。
  • Compose 点击事件中触发 Snackbar、滚动等 UI 动作用 rememberCoroutineScope。

四、调度器选择

协程调度器决定任务在哪类线程执行：

  • Dispatchers.Main：主线程，更新 UI。
  • Dispatchers.IO：网络、文件、数据库等 IO 操作。
  • Dispatchers.Default：CPU 密集型计算。
  • Dispatchers.Unconfined：特殊场景，不建议日常业务使用。

八、实践建议

协程代码建议遵守几个习惯：

  • 不在业务代码里使用 GlobalScope。
  • Repository 提供 suspend 或 Flow API。
  • ViewModel 负责启动协程。
  • 长耗时任务放到合适的 Dispatcher。
  • 不吞掉 CancellationException。
  • 并发任务优先使用 coroutineScope 或 supervisorScope。

九、小结

协程写得好不好，关键不在于用了多少 API，而在于生命周期、异常和取消是否清晰。

可以用一句话记住：任务属于作用域，失败要有边界，取消必须尊重。
